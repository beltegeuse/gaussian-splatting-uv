# Installation (uv-based)

This document describes how the project is set up under [uv](https://docs.astral.sh/uv/) using the latest Python and PyTorch versions, and how to reproduce the environment on another machine. The original conda-based `environment.yml` (Python 3.7 / PyTorch 1.12 / CUDA 11.6) is kept as a legacy reference but is no longer the recommended path.

## Resulting stack

| Component | Version |
|---|---|
| Python | 3.13 (`.python-version`) |
| PyTorch | 2.11 (cu130 wheels) |
| torchvision | 0.26 (cu130 wheels) |
| Build accelerator | ninja |
| Package manager | uv 0.11+ |

Wheel index: `https://download.pytorch.org/whl/cu130` (CUDA 13.0 binary build — forward-compatible with newer CUDA 13.x drivers).

## Hardware / driver requirements

- NVIDIA GPU with compute capability ≥ 7.0 (tested on RTX 3080, sm_86).
- NVIDIA driver providing CUDA 13.0 or newer.
- A local CUDA toolkit (nvcc) compatible with the driver, in `PATH`. The submodules are compiled from source against your local nvcc; PyTorch's cu130 wheel headers and your local nvcc must be compatible (a minor-version mismatch like 13.0 ↔ 13.2 is fine).

## One-time submodule patches

The three CUDA submodules (`simple-knn`, `fused-ssim`, `diff-gaussian-rasterization`) were written for an older PyTorch / CUDA toolchain and do not build cleanly against PyTorch ≥ 2.4 with nvcc ≥ 13. Two issues:

1. **Removed `Tensor::data<T>()` API** — PyTorch removed `tensor.data<T>()` in 2.4+; the new spelling is `tensor.data_ptr<T>()`. ~50 call sites across the three submodules.
2. **Missing `<cstdint>` includes** — `diff-gaussian-rasterization`'s headers use `uint32_t` / `uint64_t` / `std::uintptr_t` without including `<cstdint>`. Older toolchains pulled it in transitively; nvcc 13 is stricter.

The fixes are captured as portable patch files under `patches/`:

```
patches/simple-knn.patch
patches/fused-ssim.patch
patches/diff-gaussian-rasterization.patch
```

These are committed alongside the project so they travel with it.

## Reproducing the install on a fresh machine

```bash
# 1. Clone with submodules
git clone <repo-url> gaussian-splatting
cd gaussian-splatting
git submodule update --init --recursive

# 2. Apply the three submodule patches
for p in patches/*.patch; do
    name=$(basename "$p" .patch)
    git -C "submodules/$name" apply "../../$p"
done

# 3. Build & install everything
TORCH_CUDA_ARCH_LIST=8.6 uv sync
```

Replace `8.6` with the compute capability of your GPU (e.g. `7.5` for Turing, `8.9` for Ada, `9.0` for Hopper). Omitting the env var works too — it just builds for every architecture, which takes a few minutes longer.

`uv sync` reads `pyproject.toml` + `uv.lock` and:
- Picks Python 3.13 (downloading it via uv's managed Python if needed).
- Installs `torch` + `torchvision` from the cu130 index.
- Installs runtime deps (`tqdm`, `plyfile`, `opencv-python`, `joblib`, `ninja`).
- Builds the three local CUDA submodules with `--no-build-isolation` (they `import torch` at the top of their `setup.py`, so they need the venv's torch on `PYTHONPATH` at build time — `[tool.uv]` in `pyproject.toml` is configured for this).

## Verifying the install

```bash
uv run python -c "
import torch
print('CUDA:', torch.cuda.is_available(), torch.cuda.get_device_name(0))

from simple_knn._C import distCUDA2
print('simple_knn OK, shape:', tuple(distCUDA2(torch.rand(100, 3, device='cuda')).shape))

from diff_gaussian_rasterization import GaussianRasterizer
print('diff_gaussian_rasterization OK')

from fused_ssim import fused_ssim
a = torch.rand(1, 3, 64, 64, device='cuda')
b = torch.rand(1, 3, 64, 64, device='cuda')
print('fused_ssim OK, value:', fused_ssim(a, b).item())
"

uv run python train.py --help
```

All four extension imports must succeed and `train.py --help` should print the full argument list without an `ImportError`.

## Project files reference

| File | Role |
|---|---|
| `pyproject.toml` | Project metadata, dependency list, uv configuration (cu130 index, no-build-isolation packages, local submodule sources). |
| `.python-version` | Pins the Python interpreter version uv uses (`3.13`). |
| `uv.lock` | Reproducible lockfile generated by `uv sync`. **Commit this.** |
| `patches/*.patch` | Toolchain-compatibility fixes for the three submodules. **Commit these.** |
| `.venv/` | The virtualenv. Not committed; `uv sync` recreates it. |
| `environment.yml` | Legacy conda environment. Not used by uv; kept for users who prefer conda. |

## Day-to-day usage

- Add a runtime dep: `uv add <pkg>` (updates `pyproject.toml` and `uv.lock`).
- Run any project script: `uv run python <script>.py …` — uv ensures the env is in sync first.
- Re-sync after pulling new commits: `uv sync`.
- Activate the venv directly (optional): `source .venv/bin/activate`.

## Notes on the submodule patches

The three submodules are tracked as independent git submodules pointing at upstream repos. Applying the patches leaves each submodule with a dirty working tree (`git -C submodules/<name> status` will show modified files). This is expected — the upstream repos have not yet adopted these fixes. If/when the upstream maintainers merge equivalent changes, you can simply update the submodule pointer and remove the corresponding patch file.

## Troubleshooting

- **`ninja: error: subcommand failed`** during build: enable verbose output with `uv sync -v` and look at the actual nvcc error above the Python traceback.
- **`libc10.so: cannot open shared object file`** when importing a submodule: `import torch` first in the same Python process — the torch import loads the C++ runtime libraries the extensions need.
- **Mismatch warning between local nvcc and PyTorch's CUDA**: a minor-version mismatch (e.g. nvcc 13.2 vs PyTorch built against 13.0) is harmless. A major mismatch (nvcc 12 vs PyTorch cu130) will fail — install matching CUDA toolkit or switch wheel index.
